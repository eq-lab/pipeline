# Issue #50: Wire @pipeline/ui theme.css into frontend

Source: https://github.com/eq-lab/pipeline/issues/50

## Scope

Finish wiring the shared `@pipeline/ui` design tokens into `@pipeline/frontend` so the running app actually paints with the Pipeline tokens, and prove it on a placeholder route.

In scope:

- Confirm and (if needed) tighten the CSS plumbing in `packages/frontend/src/index.css` that imports `@pipeline/ui/styles/theme.css` and anchors Tailwind v4 `@source` scanning into `packages/ui/src`.
- Replace the placeholder `<main>Pipeline</main>` in `packages/frontend/src/routes/index.tsx` with a small token-styled element that exercises at least one color token, one typography token, and one radius token, sourced from `theme.css` (e.g. `bg-pipeline-paper`, `text-pipeline-ink`, `font-display`, `text-pipeline-title`, `rounded-pipeline-card`). The element is a smoke probe; it is not the final landing UI.
- Verify there is no duplicate Tailwind config / no second token source: `theme.css` in `@pipeline/ui` remains the single source of color/typography/radius tokens; `packages/frontend` has no `tailwind.config.*` and no overlapping `@theme` block.
- Regression checks: `lint`, `build`, `test`, `lint-docs`.

Out of scope:

- Adding new tokens to `theme.css` (Issue #41 is done; further token work is its own issue).

- Routing or layout changes beyond the single placeholder route at `/`.
- Backend, API, worker, or wallet integration work.

## Assumptions and Risks

- **#37 and #41 are merged.** Confirmed:
  - `packages/frontend/package.json` lists `"@pipeline/ui": "workspace:^"` under `dependencies` (#37).
  - `packages/frontend/src/index.css` already contains the three required lines (added incrementally across #37 and #40):
    ```css
    @import "tailwindcss";
    @import "@pipeline/ui/styles/theme.css";
    @source "../../ui/src/**/*.{ts,tsx}";
    ```
  - `packages/frontend/src/main.tsx` already imports `./index.css` (#37).
  - `packages/ui/src/styles/theme.css` exposes the full token set under both `:root` (raw CSS variables, post-#71 fix) and Tailwind v4 `@theme` (utility generation).
  - The `@pipeline/ui` `package.json` exports `./styles/*` so `@pipeline/ui/styles/theme.css` resolves.

  → The issue body's "Files" checklist (`main.tsx` import, `vite.config.ts` `@source`) is effectively **already done** by previous PRs. The only remaining acceptance gap is AC2: "dev server shows a token-styled element."

- **Tailwind v4 mechanism for cross-package scanning.** The issue body says "`@source` directive in `vite.config.ts` (or equivalent Tailwind v4 mechanism)." Tailwind v4's canonical mechanism is `@source` **inside CSS** (not the Vite config) — that is what is already in place. No change to `vite.config.ts` is needed and adding one would be incorrect.

- **No duplicate Tailwind config.** `packages/frontend` has no `tailwind.config.{js,ts,cjs,mjs}` file and no `@theme` block in any local CSS. The only `@theme` block in the workspace lives in `packages/ui/src/styles/theme.css`. The plan re-verifies this by grep.

- **Token usage uses Tailwind utilities, not raw `var(--…)`.** Per `docs/FRONTEND.md`: "components must not inline raw hex codes. All color, radius, and typography values are consumed through Tailwind utilities (`bg-pipeline-paper`, `text-pipeline-ink`, `rounded-pipeline-card`, `font-display`, etc.)." The probe element must use utility classes, not inline `style={{ color: 'var(--color-pipeline-ink)' }}`.

- **Where the probe lives.** Two reasonable options:
  1. Edit `packages/frontend/src/routes/index.tsx` directly (the route currently rendering `<main>Pipeline</main>`).
  2. Render a `<main>` inside `__root.tsx`.

  Adopt option (1): the issue body explicitly says "swap the placeholder route's `<main>`," which maps to `routes/index.tsx`. Leave `__root.tsx` untouched.

- **Figma reference.** The Issue links Figma node `1497-94556` (the WelcomeScreen frame). This is the **target visual language** for the eventual page, **not** the surface this Issue must reproduce. The probe is a smoke test for token plumbing, not a faithful redraw — `ux-tester` against Figma is therefore **not required** for #50; full Figma fidelity will be enforced by the component / page issues (#42–#49 and any "Welcome page" Issue). The plan still includes a visual sanity check (token color visible on screen) and a Lighthouse-free manual check, but no Figma-pixel diff is mandated.

- **TanStack file routes.** `routeTree.gen.ts` is auto-generated by the `TanStackRouterVite` plugin. Changing the route's component body does **not** require regenerating the tree (only adding/removing route files does). No `routeTree.gen.ts` edits needed.

- **`tsconfig.tsbuildinfo` is currently dirty.** The git status shows `packages/frontend/tsconfig.tsbuildinfo` modified. That is build cache noise from prior local builds; the coder should ensure the final commit does not include it (it should already be `.gitignore`-d — verify if not, log to tech-debt rather than fixing inline).

- **Risk: Tailwind v4 utility name vs token name mismatch.** `theme.css` declares e.g. `--text-pipeline-title: 64px` and `--text-pipeline-title--line-height: 64px`. The corresponding Tailwind utility is `text-pipeline-title`. The coder must confirm the utility generates the expected size+line-height pair by inspecting the dev-server DOM or built CSS. If `text-pipeline-title` does not produce the line-height variant, fall back to `text-[var(--text-pipeline-title)]` is **not** acceptable — fix the token name in `theme.css` instead and reopen #41's coverage. (This is low-risk: Tailwind v4 supports the `--text-*--line-height` paired-variable pattern natively.)

## Test Strategy

Automated (all must exit 0):

1. `yarn workspace @pipeline/frontend lint` — ESLint + Prettier; catches formatting of the new JSX.
2. `yarn workspace @pipeline/frontend build` — `tsc -b && vite build`; proves the `@import` and `@source` directives are picked up at build time, the routes file type-checks, and Tailwind v4 emits utilities for every class used in the probe (any typo'd class is a hard build error in v4 only if used inside `@apply`; for free-floating utilities Tailwind silently drops unknown classes — which is why step 5's CSS-asset grep is mandatory, not optional).
3. `yarn workspace @pipeline/frontend test` — existing Vitest run; no new tests required but must remain green.
4. `yarn workspace @pipeline/ui lint` — regression on the UI workspace.
5. `npx tsx scripts/lint-docs.ts` — required after any TS / docs change.
6. **CSS-asset content check** (step 5 above): `grep -l "pipeline-paper\|--font-display" packages/frontend/dist/assets/*.css` must find at least one file. This is the load-bearing verification that the token plumbing is actually in the shipped bundle.

Manual:

- Dev-server visual smoke (step 6 above): paper background, Besley title, bordered surface card with 4px radius.
- DevTools inspector: confirm `--color-pipeline-*` / `--font-*` custom properties resolve on the rendered elements.

Edge cases / negative checks:

- **Duplicate token source regression.** Re-grep after the edit (step 4): exactly one `@theme` block in the workspace, zero `tailwind.config.*` files.
- **Tailwind v4 silent drop.** If a probe class (e.g. `text-pipeline-title`) does not generate CSS, Tailwind v4 will emit no error — the only signal is the missing rule in the built CSS or wrong visual rendering. The CSS-asset grep + manual visual check cover this.
- **Font 404.** If `besley-regular.woff2` 404s in dev, the title will render in the system serif fallback. Log as a separate Issue (asset path / Vite font resolution from `@pipeline/ui`) — out of scope for #50 since it does not block the token-import acceptance criterion.

Figma verification: **not required for this Issue.** Per the Assumptions section, the probe is a smoke surface, not a faithful Figma redraw. `ux-tester` against Figma node `1497-94556` is deferred to the component / WelcomeScreen Issues.
